Git快速入门

发表于 2018-12-03 |

Git 的基本使用

当你在写一些长篇幅的文章时，如论文，在你对其修改一段时间后，你突然改变主意，想要之前的那个版本。而你没有保持以前的版本怎么办？是不是很蛋疼？本篇文章就带你学习版本控制工具Git来解决该问题！

简介

什么是版本控制？简单说有没有曾经写文档时有遇到这种情况

删除或修改一个段落，但又怕将来想恢复而找不回来，于是就存储了多个样本，这就是版本控制。官方的说版本控制是一种记录一个或若干个文件内容变化，以便将来查阅特定版本修订情况的系统!为什么需要它呢？人为的版本控制不是很好吗？唯一的好处就是不容易犯错，尤其是当工作目录多的时候，一不小心就会写错或者覆盖意象外的文件。所以本篇文章所介绍的这个软件，不但能自动记录每次文件的改动，还可以让同时协同编辑，这样就可以不用自己管理一堆类似的文件了。如果想要查看某次改动，只需要简单的输入几个命令就可以知道了，是不是很简单！

至于Git的历史，这里就不多介绍了，有兴趣的可以直接去Google了解一些

一次不怎么完美的邂逅

记录方式
Git对待数据的方式是直接记录快照，而非差异比较。每次提交更新，或在Git中保存项目状态时，它主要对当时的全部文件制作一个快照并保持这个快照的索引。为了高效，如果文件没有修改，Git不再重新存储该文件，而是保留一个链接指向之前存储的文件

离线运行
在Git中大多数操作都是只访问本地文件和资源，如果你用过集中式版本控制系统的话，Git在这方面会让你感到迅速与便捷。因为你在本地磁盘上就有项目完整历史，大部分操作都看起来就瞬间完成了。

保证完整性
Git中所有数据在存储前都计算校验。这意味着不可能在Git不知情的情况更改任何文件内容或目录内容

Git中使用这种哈希值的情况很多，不过这些信息都是以文件内容的哈希值来索引的，而不是文件名

三种状态

如果你需要后面的学习更顺利，请记住下面这些关于Git的概念。
Git有三种状态

  • 已提交 (committed)
    • 已提交表示数据已经安全的保存在了本地数据库中。
  • 已修改 (modified)
    • 已修改表示已经修改了文件，但还有保存到数据库中。
  • 已暂存 (staged)
    • 已暂存表示对一个已修改的文件的当前版本做了标记，使之包含在下次提交的快照中

工作目录、暂存区以及Git仓库
工作目录时对项目的某个版本独立提取出来的内容。这些从Git仓库中提取出来的软件，放在磁盘上供你使用或修改
暂存区是一个文件，保存了下次将提交的文件列表信息，有时候被称作‘索引’，不过一般说法还是叫暂存区。

  • Git工作流程
    1. 在工作目录中修改文件
    2. 添加文件到暂存区
    3. 提交更新，将暂存区中的文件永久存储到Git仓库目录，清空暂存区。
      修改了但还没有放到暂存区，就是已修改状态
      如果修改了并已放入暂存区，就属于暂存状态
      如果Git目录中保存着特定版本文件，就属于已提交状态

配置Git

Git安装完成后，你需要进行一些配置Git的环境。每台计算机上只需要配置一次，程序升级时会保留配置信息。也可以在任何时候通过运行命令来修改它们

用户信息
当用户安装完Git应该做的第一件事就是设置你的用户名与邮箱地址。因为每一次Git提交都会使用到这些信息，并且会写入你的每一次提交中，不可更改

这种配置大致分为三种情况

  1. 系统配置 (如果使用 –system 选项时使用该配置)
  2. 用户配置（如果使用 –global 选项时使用该配置）
  3. 仓库配置（针对于该仓库）

每一个级别都会覆盖上一个级别，仓库配置覆盖用户配置覆盖系统配置，一般推荐配置为 global 的。

配置方法

git 也可以设置默认的文本编辑器，当git需要你输入信息的时候会调用该编辑器，需要的可以直接去Google一下，这里就不多介绍了！

检查配置信息
配置完成后可以使用 git config --list 命令来列出所有 Git 当时能找得到的配置
也可以通过 git config 关键词来检查 git 的某一项配置

记录每次更新

现在你已经拥有了一个Git仓库了，而你需要做的就是创建一些文件，并且做些修改后，提交到本次更新到仓库
而在这个工作目录下每一个文件不外乎就两种状态

状态 描述
未跟踪 指那些被纳入了版本控制的文件，在上一次快照中有它们的记录，工作一段时间后可能处于未修改、已修改或放入暂存区状态
已跟踪 工作目录中除了已跟踪的文件都属于未跟踪文件，它们既不存在与上次快照的记录中，也没有放入暂存区。

初次克隆某一个仓库的时候，工作目录中的文件都属于已跟踪，并处于为修改状态
